| **Item** | **Details** |
|-----------|-------------|
| **Summary** | As a **service owner or API Consumer**, I want to register (and later update or deregister) **subscriber and IoT devices** as members of a **Trust Domain** — optionally supplying bootstrapping information for automated (**"zero-touch"**) onboarding — so that each device joins the segment under the Trust Domain's shared **access configuration and policies** (such as device limits, bandwidth, or egress rules), whether the goal is to onboard a brand-new device or simply to group existing devices under a common connectivity policy. |
| **Roles, Actors, and Scope** | **Roles:** `customer:user`. <br> **Actors:** Service owners, API Consumers, IoT integrators/platforms, API providers. <br> **Scope:** **Order to Activate (O2A)** — Covers registering, updating, and removing device membership in a Trust Domain and provisioning that device's network access. Device registration corresponds to TM Forum **TMF 909A § 2.1.1 "Order to Activate"** (TMF 640 Activation & Configuration), which governs service activation and configuration actions. |
| **Pre-conditions** | 1. The API Consumer has been onboarded with the API provider and obtained valid access credentials with appropriate scopes (e.g., `network-access-domains:devices`). <br> 2. A Trust Domain already exists (created via `POST /trust-domains`) into which devices can be registered. <br> 3. For automated ("zero-touch") onboarding, the device supports at least one bootstrapping protocol (DPP, Matter, or WKSO) compatible with the Trust Domain's access infrastructure. <br> 4. The API provider has implemented the **Network Access Domains** API supporting Trust Domain device registration. |
| **Begins when** | The API Consumer sends a `POST` request to `/trust-domains/{trustDomainId}/devices` with the device details (optionally including `bootstrappingInfo` and/or `deviceCredential`), or a `PATCH`/`DELETE` to `/trust-domains/{trustDomainId}/devices/{deviceId}` to update or deregister an existing device. |
| **Ends when** | The device is registered as a member of the Trust Domain (and, where bootstrapping information was provided, provisioned and associated on the infrastructure), or its membership is updated or removed, and the configuration change is successfully reflected on the associated network access devices. |
| **Post-conditions** | - The device is a member of the Trust Domain and is governed by the domain's access configuration and policies; once it completes onboarding it becomes `associated` and, while active on the network, `connected`. <br> - Updated registrations replace the prior `bootstrappingInfo` / `deviceCredential` settings (full replacement). <br> - Deregistered devices lose access to the Trust Domain. |
| **Exceptions** | - **400 Bad Request:** Invalid device payload (e.g., malformed bootstrapping URI or hardware address) or properties not declared in the specification. <br> - **401 Unauthorized:** Missing, invalid, or expired access token. <br> - **403 Forbidden:** Caller lacks sufficient authorization scope or ownership of the device or Trust Domain. <br> - **404 Not Found:** Referenced Trust Domain or device ID does not exist. <br> - **409 Conflict:** The device is already registered to the Trust Domain. |
